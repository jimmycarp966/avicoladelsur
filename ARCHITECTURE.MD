## рҹ§© Modelo de Pedidos Agrupados (Arquitectura Core)

Para optimizar la logГӯstica y la facturaciГіn, el sistema implementa un modelo de **Pedidos Agrupados**:

### Concepto: "Pedido = Ruta"
*   **Antes**: 1 Presupuesto = 1 Pedido.
*   **Ahora**: 1 Pedido es un contenedor ("bucket") que agrupa todos los presupuestos de un mismo **Turno + Zona + Fecha**.
*   **Entregas**: Los presupuestos individuales se convierten en `entregas` dentro de ese pedido maestro.

### Funcionamiento:
1.  **AgrupaciГіn AutomГЎtica**:
    *   Si un cliente hace 3 compras en la maГұana (10:00, 11:00, 12:00), todas se suman al mismo **Pedido del DГӯa**.
    *   El repartidor recibe una sola hoja de ruta consolidada.
2.  **Cierre de Pedido**:
    *   El pedido permanece "Abierto" para recibir mГЎs items hasta el **Horario de Corte** (05:00 AM para MaГұana, 15:00 PM para Tarde).
    *   Al pasar el corte, se "Cierra" y pasa a LogГӯstica.
3.  **Ventajas**:
    *   Evita duplicidad de visitas al mismo cliente.
    *   Simplifica la facturaciГіn (1 factura por turno).
    *   Optimiza la carga del camiГіn.

### 8пёҸвғЈ Dominio: Control de Sucursales вң… NUEVO (Diciembre 2025) - ACTUALIZADO (Enero 2025)

**Funcionalidades Implementadas:**

#### Control de Precios Mayorista/Minorista
- **POS Mejorado**: Componente `POSSucursal` con selecciГіn de lista de precios
- **Registro AutomГЎtico**: Cada venta registra quГ© lista se usГі (mayorista/minorista/distribuidor)
- **AuditorГӯa Completa**: Tabla `auditoria_listas_precios` con registro de todas las ventas
- **DetecciГіn de Fraude**: Alertas automГЎticas por:
  - Alto % de ventas mayoristas vs promedio (1.5x umbral)
  - Ventas mayoristas de bajo volumen (<10 kg)

#### CГЎlculo de Costo Real y Margen
- **Costo Promedio Ponderado**: FunciГіn `fn_obtener_costo_promedio_sucursal()` calcula costo real por producto
- **Margen Bruto AutomГЎtico**: Cada venta calcula margen = precio - costo
- **ValorizaciГіn de Stock**: Stock visible a costo real en tiempo real desde casa central
- **Campos Nuevos**: `detalles_pedido` incluye `costo_unitario`, `margen_bruto`, `lista_precio_id`, `tipo_lista`

#### Conteos FГӯsicos de Stock
- **Ciclo Semanal**: Sistema diseГұado para conteos fГӯsicos semanales
- **Tablas Nuevas**: `conteos_stock`, `conteo_stock_items`, `ajustes_stock`
- **Proceso Automatizado**:
  1. Iniciar conteo вҶ’ Sistema lista productos con stock teГіrico
  2. Contar fГӯsicamente вҶ’ Usuario ingresa cantidades contadas
  3. Completar conteo вҶ’ Sistema evalГәa diferencias y aplica ajustes automГЎticos
- **Tolerancia de Merma**: Configurable (default 2%)
  - Diferencias вүӨ tolerancia вҶ’ Ajuste automГЎtico (merma normal)
  - Diferencias > tolerancia вҶ’ Marca para revisiГіn (no ajusta automГЎticamente)
- **Ajustes FIFO**: Mermas aprobadas se descuentan usando FIFO (lote mГЎs antiguo primero)

#### Reportes y AuditorГӯa
- **Reporte de Uso de Listas**: Por usuario, muestra % de ventas mayorista vs minorista
- **Reporte de MГЎrgenes**: Por dГӯa y tipo de lista, muestra rentabilidad
- **Alertas de Comportamiento**: DetecciГіn automГЎtica de patrones sospechosos
- **Vista Materializada**: `mv_resumen_listas_sucursal` para reportes rГЎpidos

#### Funciones RPC Principales
- `fn_registrar_venta_sucursal()`: Venta con control completo de lista, costo y margen
- `fn_iniciar_conteo_stock()`: Inicia conteo fГӯsico
- `fn_completar_conteo_stock()`: Completa conteo y genera ajustes

### 5пёҸвғЈ Dominio: Bot de WhatsApp (Interfaz de Cliente) вң… FUNCIONANDO

**Funcionalidades Implementadas:**
- вң… **Toma de Pedido Automatizada**: ValidaciГіn de stock en tiempo real desde lotes
- вң… **Pedidos Simples**: `POLLO001 5` вҶ’ ConfirmaciГіn вҶ’ Pedido creado
- вң… **Pedidos MГәltiples**: `POLLO001 5, HUEVO001 2` вҶ’ Un solo pedido con varios productos
- вң… **ConfirmaciГіn ExplГӯcita**: Muestra resumen y requiere SГҚ/NO antes de crear
- вң… **Consulta de Estado**: `estado PED-XXXXX` o `3` para ver historial
- вң… **CatГЎlogo DinГЎmico**: Solo muestra productos con stock disponible
- вң… **Indicadores de Stock**: рҹҹў Alto (>50) | рҹҹЎ Medio (20-50) | рҹ”ҙ Bajo (<20)
- вң… **Horario de AtenciГіn**: Valida horarios laborales
- вң… **Descuento AutomГЎtico de Stock**: Sistema FIFO por lotes
- вң… **Trazabilidad Completa**: Registra quГ© lote se usГі en cada venta
- вң… **Referencias de Pago**: Genera automГЎticamente cГіdigo de referencia (formato: `PAY-YYYYMMDD-XXXXXX`) para pedidos con pago diferido
- вң… **Instrucciones para Repartidores**: Incluye monto y referencia de pago en mensajes de confirmaciГіn
- вң… **Registro de Reclamos**: Implementado (Schemas, API y Acciones)
- вң… **Registro de Clientes**: MГЎquina de estados en memoria para flujo conversacional (Nombre вҶ’ DirecciГіn вҶ’ Localidad)
- рҹҡ§ **Notificaciones Push**: En desarrollo (Implementado solo para Balanza IoT)

**MГЎquina de Estados (Registro de Clientes):**
El bot gestiona el alta de nuevos clientes mediante un flujo conversacional persistido en memoria RAM (`Map<telefono, estado>`):
1.  **Inicio**: Usuario desconocido envГӯa mensaje.
2.  **Estado 1**: `esperando_nombre` (Valida > 1 palabra).
3.  **Estado 2**: `esperando_direccion` (Valida longitud).
4.  **Estado 3**: `esperando_localidad` (SelecciГіn numГ©rica de lista dinГЎmica).
5.  **Cierre**: Crea cliente + Cuenta Corriente + Presupuesto inicial.
*Nota: Los estados expiran automГЎticamente tras 10 minutos de inactividad.*

**Sistema de Lotes (WMS Profesional):**

El bot implementa un sistema completo de gestiГіn de inventario por lotes:

1. **ValidaciГіn de Stock Real**: Consulta lotes disponibles y no vencidos
2. **FIFO AutomГЎtico**: Usa primero el lote que vence mГЎs pronto
3. **Descuento Transaccional**: Todo o nada (rollback si falla)
4. **Registro de Movimientos**: Cada venta genera movimiento de stock
5. **AsignaciГіn de Lote**: El pedido sabe de quГ© lote saliГі cada producto

**Ejemplo de flujo:**
```
Cliente pide: POLLO001 50kg

Sistema busca lotes de POLLO001 ordenados por vencimiento:
  LOTE-A: 80kg (vence 10/11) вҶҗ Se usa este
  LOTE-B: 120kg (vence 20/11)

Descuenta 50kg del LOTE-A:
  LOTE-A: 80kg вҶ’ 30kg
  
Registra movimiento:
  Tipo: salida
  Cantidad: 50kg
  Motivo: "Venta desde WhatsApp Bot"
  
Crea pedido con referencia al lote usado
```

-- FunciГіn para obtener precio de un producto en una lista especГӯfica
-- Prioridad: precio manual вҶ’ precio desde margen вҶ’ precio_venta del producto
CREATE OR REPLACE FUNCTION fn_obtener_precio_producto(
    p_lista_precio_id UUID,
    p_producto_id UUID
) RETURNS DECIMAL(10,2) AS $$
DECLARE
    v_precio DECIMAL(10,2);
    v_precio_costo DECIMAL(10,2);
    v_margen_ganancia DECIMAL(5,2);
BEGIN
    -- Buscar precio manual en la lista
    SELECT precio INTO v_precio
    FROM precios_productos
    WHERE lista_precio_id = p_lista_precio_id
      AND producto_id = p_producto_id
      AND activo = true
      AND (fecha_desde IS NULL OR fecha_desde <= CURRENT_DATE)
      AND (fecha_hasta IS NULL OR fecha_hasta >= CURRENT_DATE)
    ORDER BY fecha_desde DESC NULLS LAST
    LIMIT 1;

    -- Si no hay precio manual, calcular desde margen de ganancia
    IF v_precio IS NULL THEN
        SELECT lp.margen_ganancia, p.precio_costo
        INTO v_margen_ganancia, v_precio_costo
        FROM listas_precios lp
        CROSS JOIN productos p
        WHERE lp.id = p_lista_precio_id
          AND p.id = p_producto_id;

        IF v_margen_ganancia IS NOT NULL AND v_margen_ganancia > 0 
           AND v_precio_costo IS NOT NULL AND v_precio_costo > 0 THEN
            v_precio := v_precio_costo * (1 + v_margen_ganancia / 100);
        END IF;
    END IF;

    -- Fallback a precio_venta del producto
    IF v_precio IS NULL THEN
        SELECT precio_venta INTO v_precio
        FROM productos
        WHERE id = p_producto_id;
    END IF;

    RETURN COALESCE(v_precio, 0);
END;
$$ LANGUAGE plpgsql;

-- FunciГіn para crear pedido desde bot (atГіmica)
CREATE OR REPLACE FUNCTION fn_crear_pedido_bot(
    p_cliente_id UUID,
    p_items JSONB,
    p_observaciones TEXT DEFAULT NULL
) RETURNS JSONB AS $$
DECLARE
    v_pedido_id UUID;
    v_numero_pedido VARCHAR(50);
    v_total DECIMAL(10,2) := 0;
    v_item JSONB;
    v_stock_disponible DECIMAL(10,3);
BEGIN
    -- Generar nГәmero de pedido Гәnico
    v_numero_pedido := 'PED-' || TO_CHAR(NOW(), 'YYYYMMDD-HH24MI') || '-' || UPPER(SUBSTRING(MD5(RANDOM()::TEXT) FROM 1 FOR 4));

    -- Crear pedido
    INSERT INTO pedidos (numero_pedido, cliente_id, origen, estado, observaciones)
    VALUES (v_numero_pedido, p_cliente_id, 'whatsapp', 'confirmado', p_observaciones)
    RETURNING id INTO v_pedido_id;

    -- Procesar items
    FOR v_item IN SELECT * FROM jsonb_array_elements(p_items)
    LOOP
        -- Validar stock disponible
        SELECT SUM(l.cantidad_disponible) INTO v_stock_disponible
        FROM lotes l
        WHERE l.producto_id = (v_item->>'producto_id')::UUID
        AND l.estado = 'disponible'
        AND l.fecha_vencimiento > NOW();

        IF v_stock_disponible < (v_item->>'cantidad')::DECIMAL THEN
            RAISE EXCEPTION 'Stock insuficiente para producto %', v_item->>'producto_id';
        END IF;

        -- Insertar detalle del pedido
        INSERT INTO detalles_pedido (pedido_id, producto_id, cantidad, precio_unitario, subtotal)
        VALUES (
            v_pedido_id,
            (v_item->>'producto_id')::UUID,
            (v_item->>'cantidad')::DECIMAL,
            (v_item->>'precio_unitario')::DECIMAL,
            (v_item->>'cantidad')::DECIMAL * (v_item->>'precio_unitario')::DECIMAL
        );

        -- Actualizar total
        v_total := v_total + ((v_item->>'cantidad')::DECIMAL * (v_item->>'precio_unitario')::DECIMAL);
    END LOOP;

    -- Actualizar total del pedido
    UPDATE pedidos SET total = v_total, subtotal = v_total WHERE id = v_pedido_id;

    -- Retornar resultado exitoso
    RETURN jsonb_build_object(
        'success', true,
        'pedido_id', v_pedido_id,
        'numero_pedido', v_numero_pedido,
        'total', v_total
    );

EXCEPTION
    WHEN OTHERS THEN
        -- En caso de error, retornar el mensaje
        RETURN jsonb_build_object(
            'success', false,
            'error', SQLERRM
        );
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

-- FunciГіn para validar entrega (atГіmica)
CREATE OR REPLACE FUNCTION fn_validar_entrega(
    p_pedido_id UUID,
    p_firma_url VARCHAR(500),
    p_qr_verificacion VARCHAR(100)
) RETURNS JSONB AS $$
DECLARE
    v_ruta_id UUID;
    v_detalle_ruta_id UUID;
BEGIN
    -- Buscar el detalle de ruta correspondiente
    SELECT dr.id, dr.ruta_id INTO v_detalle_ruta_id, v_ruta_id
    FROM detalles_ruta dr
    WHERE dr.pedido_id = p_pedido_id
    AND dr.estado_entrega = 'en_camino';

    IF NOT FOUND THEN
        RETURN jsonb_build_object('success', false, 'error', 'Pedido no encontrado en ruta activa');
    END IF;

    -- Actualizar estado de entrega
    UPDATE detalles_ruta
    SET estado_entrega = 'entregado',
        fecha_hora_entrega = NOW(),
        firma_url = p_firma_url,
        qr_verificacion = p_qr_verificacion
    WHERE id = v_detalle_ruta_id;

    -- Verificar si la ruta estГЎ completa
    IF NOT EXISTS (
        SELECT 1 FROM detalles_ruta
        WHERE ruta_id = v_ruta_id
        AND estado_entrega != 'entregado'
    ) THEN
        -- Marcar ruta como completada
        UPDATE rutas_reparto
        SET estado = 'completada',
            tiempo_real_min = EXTRACT(EPOCH FROM (NOW() - (
                SELECT MIN(dr.fecha_hora_entrega)
                FROM detalles_ruta dr
                WHERE dr.ruta_id = v_ruta_id
            ))) / 60
        WHERE id = v_ruta_id;
    END IF;

    -- Actualizar estado del pedido
    UPDATE pedidos
    SET estado = 'entregado',
        fecha_entrega_real = NOW()
    WHERE id = p_pedido_id;

    RETURN jsonb_build_object('success', true, 'message', 'Entrega validada exitosamente');

EXCEPTION
    WHEN OTHERS THEN
        RETURN jsonb_build_object('success', false, 'error', SQLERRM);
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

-- FunciГіn para obtener ventas agrupadas por zona
CREATE OR REPLACE FUNCTION fn_ventas_por_zona(
    fecha_inicio DATE,
    fecha_fin DATE,
    vendedor_id UUID DEFAULT NULL
) RETURNS TABLE (
    zona VARCHAR,
    ventas DECIMAL(10,2),
    transacciones BIGINT,
    ticket_promedio DECIMAL(10,2)
) AS $$
BEGIN
    RETURN QUERY
    SELECT 
        COALESCE(c.zona_entrega, 'Sin zona') as zona,
        SUM(p.total)::DECIMAL(10,2) as ventas,
        COUNT(*)::BIGINT as transacciones,
        CASE WHEN COUNT(*) > 0 
            THEN (SUM(p.total) / COUNT(*))::DECIMAL(10,2)
            ELSE 0::DECIMAL(10,2)
        END as ticket_promedio
    FROM pedidos p
    INNER JOIN clientes c ON p.cliente_id = c.id
    WHERE p.fecha_pedido >= fecha_inicio::timestamp
      AND p.fecha_pedido <= (fecha_fin::timestamp + INTERVAL '1 day' - INTERVAL '1 second')
      AND p.estado = 'entregado'
      AND (vendedor_id IS NULL OR p.usuario_vendedor = vendedor_id)
    GROUP BY c.zona_entrega
    ORDER BY ventas DESC;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

-- FunciГіn para anГЎlisis de clientes nuevos vs recurrentes
CREATE OR REPLACE FUNCTION fn_clientes_nuevos_vs_recurrentes(
    fecha_inicio DATE,
    fecha_fin DATE
) RETURNS JSONB AS $$
DECLARE
    v_nuevos INTEGER := 0;
    v_recurrentes INTEGER := 0;
    v_total INTEGER := 0;
BEGIN
    WITH clientes_periodo AS (
        SELECT DISTINCT p.cliente_id
        FROM pedidos p
        WHERE p.fecha_pedido >= fecha_inicio::timestamp
          AND p.fecha_pedido <= (fecha_fin::timestamp + INTERVAL '1 day' - INTERVAL '1 second')
          AND p.estado = 'entregado'
    ),
    clientes_anteriores AS (
        SELECT DISTINCT p.cliente_id
        FROM pedidos p
        WHERE p.fecha_pedido < fecha_inicio::timestamp
          AND p.estado = 'entregado'
    )
    SELECT 
        COUNT(CASE WHEN ca.cliente_id IS NULL THEN 1 END)::INTEGER,
        COUNT(CASE WHEN ca.cliente_id IS NOT NULL THEN 1 END)::INTEGER,
        COUNT(*)::INTEGER
    INTO v_nuevos, v_recurrentes, v_total
    FROM clientes_periodo cp
    LEFT JOIN clientes_anteriores ca ON cp.cliente_id = ca.cliente_id;
    
    RETURN jsonb_build_object(
        'nuevos', v_nuevos,
        'recurrentes', v_recurrentes,
        'total', v_total,

### **Detalle de Optimizaciones RPC (16/12/2025)**
Para resolver problemas de rendimiento (N+1) en listados grandes, se implementaron funciones que retornan estructuras JSONB completas:

## рҹ”җ Seguridad y PolГӯticas

### Row Level Security (RLS) - Supabase
- **Principio**: Cada tabla tiene RLS activado
- **Roles del Sistema**:
  - `admin`: Acceso total a todas las funcionalidades
  - `vendedor`: GestiГіn de clientes, pedidos, cotizaciones y reclamos
  - `repartidor`: Acceso limitado a rutas asignadas y entregas
  - `almacenista`: Control de stock, ingresos y picking
- **RedirecciГіn AutomГЎtica**: Usuarios con sucursal asignada en `rrhh_empleados` son redirigidos automГЎticamente a `/sucursal/dashboard` al iniciar sesiГіn
- **PolГӯtica de Lectura de Empleados**: `empleados_read_own` permite a usuarios leer su propio registro para obtener su `sucursal_id`

### PolГӯticas de Ejemplo
```sql
-- Administradores tienen acceso total
CREATE POLICY "admin_full_access" ON pedidos FOR ALL USING (
    auth.jwt() ->> 'rol' = 'admin'
);

-- Vendedores solo pueden ver pedidos propios o de su zona
CREATE POLICY "vendedor_pedidos_zona" ON pedidos FOR SELECT USING (
    auth.jwt() ->> 'rol' = 'vendedor' OR
    cliente_id IN (
        SELECT id FROM clientes
        WHERE zona_entrega = (auth.jwt() ->> 'zona')
    )
);

-- Repartidores solo ven pedidos asignados a sus rutas
CREATE POLICY "repartidor_rutas_propias" ON pedidos FOR SELECT USING (
    id IN (
        SELECT dr.pedido_id FROM detalles_ruta dr
        JOIN rutas_reparto rr ON dr.ruta_id = rr.id
        WHERE rr.repartidor_id = auth.uid()
    )
);
```

## рҹҺҜ Modelo de Pedidos Agrupados por Turno/Zona/Fecha (Diciembre 2025)

### рҹ“Ӣ Arquitectura del Nuevo Modelo

**RevoluciГіn en el flujo operativo**: Sistema Гәnico donde un pedido agrupa automГЎticamente todas las entregas del mismo turno + zona + fecha, transformando la logГӯstica y simplificando la gestiГіn.

#### рҹҸ—пёҸ **Estructura de Datos Transformada**

##### **Nueva Tabla: `entregas`**
Cada entrega representa la porciГіn especГӯfica de un pedido para un cliente individual:

```sql
CREATE TABLE entregas (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    pedido_id UUID NOT NULL REFERENCES pedidos(id) ON DELETE CASCADE,
    cliente_id UUID NOT NULL REFERENCES clientes(id),
    presupuesto_id UUID REFERENCES presupuestos(id),

    -- Totales por entrega
    subtotal DECIMAL(12,2) NOT NULL DEFAULT 0,
    recargo DECIMAL(12,2) DEFAULT 0,
    total DECIMAL(12,2) NOT NULL DEFAULT 0,

    -- UbicaciГіn histГіrica
    direccion TEXT,
    coordenadas GEOMETRY(POINT, 4326),

    -- Estados de entrega
    orden_entrega INTEGER,
    estado_entrega VARCHAR(50) DEFAULT 'pendiente',
    fecha_hora_entrega TIMESTAMPTZ,
    notas_entrega TEXT,
    firma_url VARCHAR(500),
    qr_verificacion VARCHAR(100),

    -- Estados de pago individuales
    estado_pago VARCHAR(20) DEFAULT 'pendiente',
    metodo_pago VARCHAR(50),
    monto_cobrado DECIMAL(12,2) DEFAULT 0,
    numero_transaccion VARCHAR(100),
    comprobante_url VARCHAR(500),
    notas_pago TEXT,
    pago_validado BOOLEAN DEFAULT false,

    -- Referencia de pago Гәnica
    referencia_pago VARCHAR(60),
    instruccion_repartidor TEXT,

    -- Observaciones del presupuesto
    observaciones TEXT,

    -- AuditorГӯa
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

##### **Pedido Transformado**
- **`cliente_id` ahora opcional**: Ya no hay "cliente del pedido"
- **Nuevo campo `estado_cierre`**: 'abierto' (agrupando) / 'cerrado' (completado)
- **Nuevo campo `hora_cierre`**: Timestamp automГЎtico cuando se cierra
- **Nuevo campo `cantidad_entregas`**: Conteo de entregas en el pedido
- **ГҚndice Гәnico condicional**: Solo pedidos abiertos tienen restricciГіn por turno/zona/fecha
